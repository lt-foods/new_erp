## Q1 — 美食列車是 module 還是 operational concept？

**決議：operational concept，不開 module。**

**理由：**
- lt-erp audit 沒找到對應 module，實務上只是「0 天保存、當日清完」的存貨特性
- `products` / `skus` 已有 `storage_type` 欄位可涵蓋（`frozen / chilled / ambient / food_truck`）
- 搭配 `retention_days = 0` 就能讓庫存 / 採購 / 通知三個模組自然 catch
- 開獨立 module 反而複雜化：business logic 分散、另設 RLS policy

**影響：**
- 商品 PRD 無需改動（`storage_type` enum 已含 `food_truck`）
- 後續若真的出現跨 module 的列車專用流程，才考慮升格 module

## Q2 — BUG-014（lt-erp `generateSalesOrder` 有 0 detail）根因？

**決議：不追 lt-erp 根因。**

**理由：**
- 使用者明確表示「這不是 new_erp 的 bug」
- new_erp 已 from-scratch 重寫，data 層也避開 lt-erp 的 `shared_kv` jsonb 整檔覆蓋 race
- 花時間還原 lt-erp bug 對 new_erp 沒有直接價值

**應用：**
- `generate_sales_order_from_wave` RPC **保守防禦**：
  - `pg_advisory_xact_lock(wave_id)` 序列化同 wave 的呼叫
  - 寫一筆前先 `SELECT ... FOR UPDATE` 鎖 wave row
  - 有 0 item 直接 `RAISE EXCEPTION`，不默默 commit
- **不預設** lt-erp 的 root cause（避免帶進錯誤假設）
- 反模式列於 integration plan §4 #4，實作時對照

## Q3 — Apps Script repo 配置？

**決議：monorepo `apps-script/`（放 new_erp repo 內）。**

**理由：**
- Apps Script 程式（樂樂 CSV / xiaolan sync / 1688 / 拼多多 import）與 Supabase schema 高度耦合
- 放獨立 repo（`new_erp_apps_script`）會讓 schema 改動與 Apps Script 改動跨 repo，PR / review 切裂
- lt-erp 目前也是 repo 內路徑（參考值：操作習慣對齊）

**應用：**
- Repo 結構：
  ```
  new_erp/
  ├── apps-script/
  │   ├── xiaolan-sync/         # 6 張 xiaolan_* 表同步
  │   ├── lele-csv-import/      # 樂樂通路 CSV
  │   ├── marketplace-import/   # 1688 / 拼多多
  │   └── README.md
  ├── docs/
  ├── supabase/
  └── ...
  ```
- 每個子 Apps Script 用 `clasp` 管理部署（每個子資料夾一個 `.clasp.json`）
- PR 規則：Apps Script 程式碼改動 + schema 改動 **同一個 PR**，避免漂移

- 若 v1 只用 Excel 頂著，月結手動對帳會反咬 pilot（lt-erp 現況就是這樣）
- 6–8 張新表 + 3 個 RPC 是可負擔的增量

  - `expenses` / `expense_categories`
- RPC：`web_make_payment` / `web_add_expense` / `web_approve_expense`
- 會計流程對齊 lt-erp UX、schema 從零設計（不照搬 shared_kv）

## Q5 — 退回龍潭 vs 店轉店 分合？

**決議：enum 合併，UX 兩按鈕 / data 單表。**

**理由：**
- `transfers` 表已存在、欄位足以涵蓋「A → B」的通用轉移語意
- 加 `transfer_type` enum（`'store_to_store' | 'return_to_hq' | 'hq_to_store'`）就能區分業務流
- data 單表 → RLS policy 一套、report / aging 一套、加盟主月結 netting 一次搞定
- UX 層兩個按鈕（「店轉店」vs「退回龍潭」）只是 client-side 預填不同 defaults

**應用：**
- Schema delta（v0.2 Phase 3.5）：
  ```sql
  ALTER TABLE transfers
    ADD COLUMN transfer_type TEXT NOT NULL DEFAULT 'store_to_store'
    CHECK (transfer_type IN ('store_to_store', 'return_to_hq', 'hq_to_store'));
  ```
- 庫存 PRD v0.2 addendum 詳載 UX 雙按鈕設計
- 採購 PRD v0.2 addendum 補「退回龍潭」在 GR 退貨流程中的觸發點
